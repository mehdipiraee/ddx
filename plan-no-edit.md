# Plan: Single source of truth — beads owns plan content, plan.md is dashboard-only

## Context

When beads tracking is enabled, plan details end up in two places: `plan.md` (full content during Generate/Review) and beads tasks (after sync). Users can edit `plan.md` directly, causing it to desync from beads. The fix: when beads is enabled, `plan.md` should **never** hold the full plan content — beads is the sole source of truth.

## Approach

When beads is enabled, present the plan **inline in the conversation** during Generate/Review instead of writing it to `plan.md`. After approval, beads tasks are created and `plan.md` is written as a dashboard-only file (by the hook). This means `plan.md` never contains plan details when beads is active.

## Changes

### 1. `skills/ddx.plan/SKILL.md` — Generate section

In both Product Plan and Capability Plan subsections, add a beads-enabled branch:

- **Beads enabled**: Generate the plan content but present it **inline in the conversation** as formatted markdown. Do NOT write it to `plan.md`.
- **Beads NOT enabled**: Write to `plan.md` as before (current behavior).

### 2. `skills/ddx.plan/SKILL.md` — Review section

Update the review instructions for the beads-enabled case:

- **Beads enabled**: Tell the user to review the plan shown in the conversation. If they want changes, make them and re-present inline. No file to review.
- **Beads NOT enabled**: Same as today — "review the file at `{output path}`".

### 3. `skills/ddx.plan/SKILL.md` — Beads Task Sync section

Remove step 2 ("Re-read the plan file from disk. Parse all steps.") — when beads is enabled, the plan content is already in the conversation context from the Generate phase, not in plan.md. Parse from what was generated, not from a file.

### 4. `skills/ddx.build/SKILL.md` — Load Beads Status fallback (step 3)

The current fallback says: "If query returns zero tasks, re-read `plan.md` and create beads tasks from it." Since `plan.md` won't have content when beads is enabled, change this to:

- Read the upstream docs (definition, spec, design) and generate the plan steps from them (same as `/ddx.plan` would). Then create beads tasks. This is essentially running the plan generation + beads sync inline.

### 5. `hooks/ddx-refresh-plan.sh` — Dashboard output

Add a "Do not edit this file" warning to the generated dashboard. Change the blockquote to:

```
> Auto-generated from Beads. If you edit this file, it will get out of sync with Beads.
> Source of truth: `bd list --label {label}`
> Last refreshed: {timestamp}
```

## Files to modify
